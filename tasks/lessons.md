# 작업 교훈

## 권리

- 성경이라는 이유만으로 모든 번역문이 자유 이용 가능한 것은 아니다.
- 공개 사이트의 기본 코퍼스는 제공처가 Public Domain을 명시한 판본으로 제한한다.
- 현대 번역본은 코드와 분리하고 서면 허락 전에는 공개 저장소에 넣지 않는다.

## 검색

- 한국어 성경 검색은 본문뿐 아니라 책 이름·약칭·영문 코드·장·절 표기를 함께 색인해야 한다.
- 질문형 입력의 조사와 일반 질문어를 제외해야 핵심 본문 단어가 묻히지 않는다.
- 직접 구절 표기는 키워드 점수보다 우선해서 정확히 일치시킨다.
- 정적 사이트의 HTML만 갱신해도 브라우저가 같은 URL의 CSS·검색 모듈·본문 JSON을 오래 재사용할 수 있다. 배포 단위 버전 쿼리를 모든 핵심 자산에 함께 붙인다.
- 무작위 구절은 본문 로드 전 버튼을 비활성화하고 직전 ID를 피해야 빈 이동과 연속 중복을 줄일 수 있다.

## RAG 운영

- Haystack의 파이프라인은 Python 실행 환경이 필요하므로 순수 정적 Cloudflare Pages 안에서 직접 실행하지 않는다.
- 운영 백엔드를 먼저 늘리지 말고 네이티브 문서 내보내기와 평가 질의로 검색 품질을 확인한 뒤 희소·밀집 하이브리드 검색을 도입한다.
- 메모리 문서 저장소와 BM25는 재현 가능한 로컬 기준선으로만 쓰고, 운영 영구 저장소로 오인하지 않는다.
- 한국어 질문에서 단순 BM25 순위는 형태 변화에 취약하므로 Vertex 다국어 임베딩을 더 높은 가중치로 결합하고 근거 원문을 함께 노출한다.
- 문서 임베딩은 빌드마다 다시 계산하지 않고 문서 ID와 1:1로 정렬한 결정적 산출물로 보존해야 콜드 스타트와 API 비용을 줄일 수 있다.
- `요 3:16` 같은 정확한 위치 표기는 임베딩만으로 찾지 않는다. 브라우저와 같은 책 약칭 파서를 백엔드에도 두고 해당 절을 포함한 passage를 RAG 근거 1순위로 고정한다.

## 원본 검수

- 제공처가 전체 성경으로 배포해도 실제 장·절 수를 별도로 검증한다.
- eBible.org의 Korean Bible 1910 VPL과 USFM은 모두 베드로전서가 4장 14절에서 끝난다.
- 상류 원본의 누락을 임의의 다른 번역으로 메우지 않고 QA 문서와 화면에 밝힌다.
- eBible.org의 WEB-C 2026-05-22 VPL ZIP은 창세기 1,533절이 빠져 있으므로 같은 배포본의 USFM과 책·장·절 수를 교차 검증한다.
- 73권이라는 책 수만 맞아도 충분하지 않다. 요엘·말라키처럼 판본별 장절 구분이 달라서 가톨릭식 장 수와 경계 절을 별도 검사해야 한다.

## 기계 번역

- 성경 기계 번역은 장문 한 번보다 장 경계를 지킨 소규모 절 묶음과 절 ID 완전 일치 검사가 안전하다.
- 책별 JSONL 체크포인트에 성공한 묶음을 즉시 기록해 GPU·네트워크 중단 뒤에도 재개 가능하게 한다.
- 기계 번역문은 공인 번역으로 오인되지 않도록 출처, 모델, 프롬프트 버전과 비공인 초안 고지를 함께 보존한다.
- 한국어 출력 여부만 검사하면 `공虛`나 영문 고유명사 같은 혼용을 놓친다. 라틴·그리스·키릴·히브리·아랍·한자·일본 문자 범위와 개신교식 핵심 용어도 체크포인트 저장 전에 거부한다.
- 희귀 인명 한 절의 반복 실패가 전체 번역을 멈추게 해서는 안 된다. 성공 절은 저장하고 실패 절은 별도 큐로 격리해 나머지 코퍼스를 계속 처리한다.
- 대표 구절만 교정에 통과해도 조사·서술어 활용과 오타가 남을 수 있다. 2차 교정 프롬프트에 실제 실패 유형을 예시로 넣고, 교정 결과도 전체 절 수와 금지 문자로 다시 검사한다.
- JSON 복구를 반복해도 같은 절이 실패하면 형식 지시와 번역 지시가 충돌한 것일 수 있다. 소수 잔여 절은 같은 모델에 본문 한 줄만 요청하고 동일한 문자·용어 검사를 적용한다.
- 장 단위로 같은 오류가 몰리는데 실행 중인 프로세스가 개선 전 복구 로직을 쓰고 있다면 호출을 계속 낭비하지 않는다. 체크포인트를 확인하고 중단한 뒤 개선된 로직으로 재개한다.
- 관리형 모델 요청이 장시간 모두 멈추면 5분짜리 기본 타임아웃을 반복하지 않는다. 체크포인트에서 재개하고 동시 요청 수와 요청 타임아웃을 낮춰 장애 반경을 줄인다.

## 재현 가능한 빌드

- 문서·PDF 빌드는 시스템 `python3`에 의존하지 않는다. `python-docx`·ReportLab·pypdf가 함께 설치된 동일한 런타임으로 생성과 검증을 끝까지 실행한다.
- CI가 성공해도 액션 런타임 폐기 경고는 남을 수 있다. GitHub 공식 액션의 최신 릴리스를 확인해 메이저 태그를 갱신하고 경고 없는 재실행까지 확인한다.
- ZIP 생성 시 기본 파일 시간값을 쓰면 내용이 같아도 CI에서 바이너리 차이가 발생한다.
- ZIP 엔트리의 시간과 권한을 고정하고, 연속 빌드의 SHA-256이 같은지 검증한다.
- DOCX도 내부적으로 ZIP이므로 문서 속성 시간을 고정해도 패키지 엔트리 시간이 매번 달라진다. 저장 뒤 ZIP 시간·권한을 정규화해야 생성물 차이 검사가 안정적이다.
- 한글 PDF는 독자의 컴퓨터에 글꼴이 없어도 보이도록 TTF를 파일에 포함한다. CID 글꼴은 렌더러에 CMap이 없으면 빈 페이지처럼 보일 수 있다.
- DOCX는 XML에 한글 글꼴 이름만 넣고 끝내지 않는다. LibreOffice 렌더에서 표지·중간·마지막 페이지를 확인해 실제 글꼴 대체 실패를 잡는다.
- 긴 읽기용 PDF는 73권 목차와 책갈피를 자동 생성하고, 절마다 새 줄을 시작해야 탐색성과 가독성이 함께 좋아진다.

## 음성 읽기

- Web Speech API의 음성 수와 품질은 운영체제·브라우저에 따라 달라지므로 특정 목소리를 보장하지 않는다.
- 화면 잠금 중 TTS 지속 재생은 모바일 브라우저 정책에 따라 멈출 수 있어 화면과 FAQ에 한계를 밝힌다.
- 외부 TTS API 없이 기기 음성을 우선 사용해 키 관리, 사용료, 본문 전송 부담을 없앤다.
- 발화 내용 회귀 검사는 화면의 현재 절 표시까지 막지 말고 `SpeechSynthesisUtterance`에 전달되는 문자열만 검사한다.
- 장별 MP3는 절 번호를 원고 단계에서 제외하고, 짧은 TTS 조각을 생성한 뒤 컨테이너의 ffmpeg로 합쳐 API 요청 한도와 장 단위 재생을 함께 만족시킨다.
- 로컬에 ffmpeg가 없어서 병합이 실패한 경우를 TTS 실패로 오인하지 않는다. 한 조각 MP3와 ffmpeg가 포함된 실제 컨테이너 Job을 각각 검증한다.
- 오디오 파일은 Cloud Run이 중계하지 않고 공개 읽기 전용 Cloud Storage에서 Range 응답으로 직접 스트리밍해 비용과 장애 지점을 줄인다.
- `gcloud run jobs deploy --args`에서 한 인자 안의 쉼표는 별도 인자로 분리된다. `--voices=kore,charon`처럼 쉼표를 보존해야 할 때는 gcloud의 대체 구분자 문법을 사용하고 배포된 Job 인자를 다시 읽어 검증한다.
- 대량 Chirp 합성에서는 일부 요청이 일시적으로 `500/503`을 반환할 수 있다. 장 전체를 즉시 실패시키지 말고 TTS 조각 단위 지수 백오프를 적용하며, 완성 객체는 건너뛰는 재실행으로 누락만 복구한다.
- Chirp 3 기본 한도는 프로젝트당 분당 200요청이므로 병렬 작업 수는 이 한도와 실제 응답 시간을 함께 보고 정한다. 짧은 작업을 더 많은 파티션으로 나누면 한 작업의 타임아웃 위험도 줄일 수 있다.
- 같은 본문이 `asia-northeast1` 엔드포인트에서 반복 `500`이고 global에서 연속 성공하면 본문 오류가 아니라 리전 장애로 판단한다. 소량의 동일 입력 A/B 테스트 뒤 안정적인 global 엔드포인트로 전환한다.
- Chirp는 줄바꿈만으로 문장 경계를 인식하지 않아 마침표 없는 절이 연속되면 `sentence too long`을 낼 수 있다. 절 번호는 빼되 각 절 발화 끝에는 문장 종결 부호를 보장한다.

## GCP 운영

- Cloud Run의 서비스 전체 `--max`와 리비전 `--max-instances`는 별도 설정이다. 비용 상한을 명확히 유지하려면 둘을 같은 값으로 배포하고 서비스·템플릿 annotation을 모두 확인한다.
- 운영 JSON을 `curl | python -c`로 검사할 때 중첩 f-string 따옴표를 쓰면 쉘 이스케이프 오류가 API 실패처럼 보일 수 있다. 단순 `format`이나 검증 스크립트를 사용한다.
